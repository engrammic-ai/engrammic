# v1.5 Agent Identity Inheritance: Risk Matrix & Mitigations

**Document**: Risk assessment for v1.5 feature implementing agent identity inheritance in context-service.

**Feature Goal**: Enable agents (Agent B) spawned from parent agents (Agent A) to inherit A's identity context, allowing reflections to reference parent agent knowledge within epistemological boundaries.

**Scope**: Memgraph model changes, AuthContext enrichment, MCP tool surface, graph query complexity, migration strategy.

**Date**: 2026-05-03  
**Status**: Assessment phase

---

## Executive Summary

Agent identity inheritance introduces moderate risk across three dimensions:

1. **Data model**: Graph traversal for lineage is tractable; silo-scoped tenancy boundary is solid.
2. **Auth & MCP**: Current `agent_id` is optional and weakly typed; inheritance requires parent tracking and schema changes.
3. **Migration**: Existing systems (no parent tracking) can coexist; opt-in adoption minimizes churn.

**Key Finding**: Risk is manageable if inheritance is opt-in and queries are bounded by depth + silo. Breaking changes are avoidable.

---

## 1. TECHNICAL RISKS

### 1.1 Graph Traversal Complexity & Performance

**Risk**: Agent lineage chains grow unbounded; traversal for deep inheritance causes N+1 queries or runaway Cypher complexity.

**Likelihood**: Medium (depends on agent deployment patterns)  
**Severity**: High (impacts `context_reason`, `context_reflect` p95 latency targets)

**Mitigations**:
- **Bounded depth**: Cap lineage traversal at 3 hops (Agent A <- B <- C <- D stops at D, omits E). Justify in ontology: inheritance is epistemically local, not recursive reasoning.
- **Index on parent**: Create `CREATE INDEX ON :Agent(parent_id)` + `CREATE INDEX ON :Agent(silo_id)` for lineage queries.
- **Cached lineage**: Redis key `agent:lineage:{silo_id}:{agent_id}` stores ancestors as JSON list, TTL 1h. Refresh on new parent edges.
- **Query plan**: `MATCH (a:Agent {id: $id, silo_id: $silo})-[:SPAWNED_FROM*0..3]->(ancestor:Agent) RETURN ancestor.id AS id ORDER BY ancestor.id` — explicit bound prevents runaway.
- **Threshold metric**: Monitor query time in logs; alert if lineage queries exceed 50ms p95.

**Blockers**: None if bounded depth is enforced at query time, not data design time.

---

### 1.2 Identity Resolution Ambiguity

**Risk**: A reflection claims to use parent A's knowledge, but A's identity is stale, deleted, or collides with another agent (name collision in logs).

**Likelihood**: Medium-high  
**Severity**: Medium (causes confusion in audit trails, not data loss)

**Mitigations**:
- **Deterministic agent IDs**: Agent IDs are UUIDs, not user-provided names. No collision.
- **Immutable parent snapshot**: Store parent identity at edge creation time: `(b)-[:SPAWNED_FROM {parent_id: uuid, parent_silo_id: uuid, spawned_at: timestamp}]->(a)`. If A is later deleted, edge retains historical reference.
- **Orphan handling**: Queries that fail to resolve a parent fall back gracefully (treat as if no parent, log warning). No reflection is rejected.
- **Validation on link**: `MATCH (b:Agent {id: $b_id, silo_id: $silo}), (a:Agent {id: $parent_id, silo_id: $silo}) CREATE (b)-[:SPAWNED_FROM {...}]->(a)` ensures parent exists at link time.

**Blockers**: None if immutability of the edge is enforced at schema level.

---

### 1.3 Data Model Brittleness

**Risk**: Adding `:SPAWNED_FROM` edge and `parent_id` property creates two sources of truth; inconsistency between them breaks queries.

**Likelihood**: Low  
**Severity**: Medium (queries fail or return incomplete lineage)

**Mitigations**:
- **Single source of truth**: Store parent identity ONLY on the edge, not as a property on the node. This enforces single query path: `(b)-[:SPAWNED_FROM]->(a)`.
- **Edge properties only**: `SPAWNED_FROM` edges carry `parent_id`, `parent_silo_id`, `spawned_at`. Never duplicate these on the Agent node.
- **Cypher validation**: All lineage queries use `MATCH (a:Agent)-[:SPAWNED_FROM*0..3]->(ancestor:Agent)`, never `a.parent_id`. Code review enforces this.
- **Schema index**: `CREATE INDEX ON :Agent(id, silo_id)` — uniqueness on this pair ensures no duplicate agents within a silo.

**Blockers**: None if edge-only storage is documented and code-reviewed.

---

### 1.4 Silo Boundary Leakage

**Risk**: Inheritance allows cross-silo identity lookups; a bug in lineage queries allows Agent B (silo X) to inherit from Agent A (silo Y).

**Likelihood**: Low  
**Severity**: Critical (multi-tenant data breach)

**Mitigations**:
- **Mandatory silo check**: Every lineage query includes `WHERE ancestor.silo_id = $silo_id` at each hop. Example:
  ```cypher
  MATCH (b:Agent {id: $id, silo_id: $silo})-[:SPAWNED_FROM*0..3]->(ancestor:Agent)
  WHERE ancestor.silo_id = $silo
  RETURN ancestor.id
  ```
- **Constraint at edge creation**: `CREATE (b:Agent {silo_id: $silo1})-[:SPAWNED_FROM]->(a:Agent {silo_id: $silo2})` is rejected by the database if silo_ids differ. Add Cypher guard: `WITH a WHERE a.silo_id = b.silo_id OR THROW`.
- **Test coverage**: Add test_silo_boundary_enforcement.py with attempts to create cross-silo SPAWNED_FROM edges — must fail.
- **Audit log**: Every failed cross-silo link attempt is logged with org_id, agent_ids, timestamp. Monitor for pattern anomalies (potential attack).

**Blockers**: None if constraint is enforced at Cypher execution time before edge is created.

## 2. IMPLEMENTATION RISKS

### 2.1 AuthContext Schema Extension

**Risk**: Adding `parent_agent_id` to `AuthContext` breaks compatibility with existing WorkOS sealed-session format; tokens fail verification.

**Likelihood**: Medium (new field must be optional)  
**Severity**: High (auth failures block all MCP calls)

**Mitigations**:
- **Backward-compatible AuthContext**: Add optional field `parent_agent_id: str | None = None` to `AuthContext` (frozen dataclass). Existing tokens without this field remain valid.
- **WorkOS metadata layer**: Store parent ID in WorkOS session metadata (custom claims), not in base token. Resolver reads it: `session.metadata.get("parent_agent_id")`.
- **No changes to bearer token format**: Sealed-session token remains opaque. Only `resolve_mcp_auth_from_header` (in auth/resolve.py) extracts parent_agent_id from WorkOS metadata.
- **Gradual rollout**: Deploy with `parent_agent_id = None` default for all existing sessions. Only new parent-child spawns populate it.
- **Smoke test**: Run integration tests against staging WorkOS to verify token verification remains < 50ms.

**Blockers**: Requires WorkOS custom claims to be enabled and tested before merge.

---

### 2.2 Migration of Existing Agent References

**Risk**: Existing nodes reference `agent_id` as a property (e.g., `:ReasoningChain {produced_by_agent_id}`, `:MetaObservation {agent_id}`). New parent-child queries don't find these agents because they have no `:Agent` node in the graph.

**Likelihood**: High (live data exists)  
**Severity**: Medium (inheritance doesn't work for historical agents, but doesn't break reads)

**Mitigations**:
- **Lazy Agent creation**: On first reference to an `agent_id`, create an `:Agent` node if it doesn't exist. Query becomes:
  ```cypher
  MERGE (a:Agent {id: $agent_id, silo_id: $silo})
  MATCH (a)-[:SPAWNED_FROM*0..3]->(ancestor:Agent)
  WHERE ancestor.silo_id = $silo
  RETURN ancestor.id
  ```
  MERGE ensures the agent is present before lineage traversal.
- **Backfill script**: One-time migration: for all unique `(produced_by_agent_id, silo_id)` pairs in ReasoningChain, create `:Agent` nodes. Opt-in via `just backfill-agent-nodes`.
- **No external IDs needed**: Agent nodes use the same `agent_id` string as properties. No mapping table.
- **Idempotency**: Backfill script is idempotent (MERGE, not CREATE). Safe to re-run.

**Blockers**: None if MERGE is used in all lineage queries.

---

### 2.3 MCP Tool Surface Changes

**Risk**: Adding `parent_agent_id` parameter to `context_reason` and `context_reflect` breaks existing clients; client auth headers must carry the field.

**Likelihood**: Medium  
**Severity**: Low (new parameter is optional)

**Mitigations**:
- **Opt-in parameter**: Add `parent_agent_id: str | None = None` to MCP tool signatures (context_reason, context_reflect, context_remember). Clients that don't provide it default to None.
- **Auth fallback**: If parameter is missing, try to read `parent_agent_id` from `AuthContext` (derived from WorkOS metadata). If still None, agent acts as root (no inheritance).
- **No breaking changes**: Existing clients omit the parameter; they continue to work.
- **Documentation**: Update context/api-examples.md with a "spawning agents" example showing how to pass parent_agent_id.

**Blockers**: None if parameters are optional and documented.

---

### 2.4 Reflection Logic Changes

**Risk**: `context_reflect` currently reads claims from only the calling agent's silo. With inheritance, reflections on parent claims require cross-agent visibility; permission boundary becomes unclear.

**Likelihood**: Medium  
**Severity**: High (security/privacy issue if scoping is wrong)

**Mitigations**:
- **Inheritance-only visibility**: Child agent can read/reflect on parent's claims ONLY if the parent_agent_id lineage is verifiable (edge exists in graph). Not allowed to hardcode parent IDs.
- **Visibility rule**: `MATCH (child:Agent {id: $agent_id, silo_id: $silo})-[:SPAWNED_FROM*0..3]->(parent:Agent) ... MATCH (parent)-[:PRODUCED]->(claim)` — claims are only readable if parent is reachable.
- **Audit log**: Every reflection on an inherited claim logs `parent_agent_id` and the edge distance (hop count). Monitor for unexpected inheritance patterns.
- **Test: test_reflection_inheritance_boundary.py**: Assert that a child cannot reflect on sibling or unrelated claims.

**Blockers**: None if visibility is gated by edge traversal.

## 3. OPERATIONAL RISKS

### 3.1 Identity Sprawl

**Risk**: Agents proliferate without lifecycle management; lineage chains grow unbounded, consuming storage and making traversal slow.

**Likelihood**: High (AI agents are cheap to instantiate)  
**Severity**: Medium (disk/memory cost, not correctness)

**Mitigations**:
- **Soft limit on depth**: Queries cap at 3 hops; agents beyond that are unreachable for inheritance. Document as: "inheritance is local, not recursive."
- **Cleanup policy**: Agents with `created_at < now - 90 days` and no incoming edges (no children, no claims) are eligible for archival. Not deleted; archived to a separate `:ArchivedAgent` node. Lineage queries skip them.
- **Metrics**: `gauge: graph.agent_count (silo_id)`, `gauge: graph.max_lineage_depth (silo_id)`. Alert if agent count exceeds 10k/silo or max depth exceeds 5.
- **Documentation**: Add CLAUDE.md note: "Agent nodes are not auto-deleted; use admin API to archive stale agents."

**Blockers**: None if metrics are in place.

---

### 3.2 Orphaned Identities

**Risk**: Parent agent is deleted/archived, but children and their claims still reference the (now inaccessible) parent. Claims are valid but attribution is lost.

**Likelihood**: Medium  
**Severity**: Low (audit trail clarity issue, not data loss)

**Mitigations**:
- **Immutable parent snapshot**: The SPAWNED_FROM edge stores parent metadata at creation time (`parent_id`, `parent_silo_id`, `spawned_at`). Even if parent node is deleted, the edge preserves the fact that a relationship existed.
- **Soft delete**: Mark agents as deleted with a property `deleted_at: timestamp` instead of hard deletion. Lineage queries can skip deleted ancestors if needed: `WHERE ancestor.deleted_at IS NULL`.
- **Orphan queries**: Add a read-only endpoint (admin) to find orphaned children: `MATCH (child:Agent)-[:SPAWNED_FROM]->(parent:Agent) WHERE parent.deleted_at IS NOT NULL RETURN child.id, parent.id`.
- **Retention policy**: Deleted agents are retained for 2 years for audit. After that, they're purged.

**Blockers**: None if soft delete is used.

---

### 3.3 Cross-Silo Leakage via Misconfiguration

**Risk**: A misconfigured MCP call passes `parent_agent_id` from a different org/silo; the query accidentally returns claims from the wrong tenant.

**Likelihood**: Low (silo checks are in place)  
**Severity**: Critical (data breach)

**Mitigations**:
- **Authorization guard**: Every MCP tool checks that parent_agent_id (if provided) belongs to the same silo as the calling agent. Example in `context_reason`:
  ```python
  if parent_agent_id:
      parent = await store.get_agent(parent_agent_id, silo_id)
      if parent is None or parent.silo_id != silo_id:
          raise ValueError(f"parent_agent_id {parent_agent_id} not found in silo {silo_id}")
  ```
- **Test: test_cross_silo_parent_rejection.py**: Attempt to set parent_agent_id to an agent from a different silo. Must fail with clear error.
- **Audit log**: Log every parent_agent_id lookup with org_id, silo_id, parent_id, result (found/not found). Monitor for anomalies.

**Blockers**: None if authorization guard is added to all MCP tool entry points.

## 4. MIGRATION & ROLLOUT RISKS

### 4.1 Backward Compatibility

**Risk**: Queries change (new SPAWNED_FROM edge, new Agent nodes); old clients/tests break.

**Likelihood**: Medium  
**Severity**: Low (tests can be updated)

**Mitigations**:
- **Feature flag**: Add `ENABLE_AGENT_IDENTITY_INHERITANCE` to settings.py (default False). MCP tools and queries only check lineage if enabled.
- **Soft launch**: Deploy with flag off. Run in shadow mode (log what would happen, don't change behavior). Gather metrics on lineage depth, agent count, etc.
- **Gradual opt-in**: Enable for specific orgs/silos in staging, then canary in prod (10% of traffic).
- **Rollback plan**: If issues arise, disable flag and revert MCP tool logic to simple `agent_id` (no inheritance). No DB migrations needed because SPAWNED_FROM edges are new (no existing data to revert).

**Blockers**: None if feature flag is in place.

---

### 4.2 Data Backfill for Existing Agents

**Risk**: Existing system has MetaObservation, ReasoningChain nodes with `agent_id` property, but no :Agent nodes. Backfill script fails or is slow.

**Likelihood**: Medium  
**Severity**: Medium (backfill takes time, blocks enablement)

**Mitigations**:
- **Idempotent backfill**: Script uses MERGE, not CREATE. Safe to re-run.
- **Batching**: Process agents in chunks of 1000 to avoid lock contention. Use `SKIP 0 LIMIT 1000` pattern.
- **Dry run**: First pass collects stats (how many agents, how many nodes reference them). Prints to stdout, doesn't modify DB.
- **Progress monitoring**: Script logs `backfill_progress: {completed}/{total}` every 1000 agents. Can be paused and resumed from checkpoint.
- **Validation**: After backfill, run `test_agent_backfill_completeness.py` to ensure all agent_id references have corresponding :Agent nodes. If not, backfill again.

**Blockers**: None if backfill is tested in staging first.

3. Implement bounded lineage query in `engine/protocols.py`: `get_agent_lineage(agent_id, silo_id, max_depth=3)`.

## 7. SUCCESS CRITERIA

1. **No performance regression**: Lineage queries remain < 50ms p95 across all test scenarios.
2. **No data leakage**: Zero cross-silo inheritance across 100 test orgs.
3. **Backward compatible**: Existing agents without parents work unchanged; inheritance is opt-in.
4. **Audit trail complete**: Every inherited claim reference logged with hop count and timestamp.
5. **Rollback possible**: Feature flag allows instant disable with no data migration.

## 8. OPEN QUESTIONS / DEFERRED DECISIONS

1. **Inheritance visibility for commitments**: Should `context_commit` (Wisdom layer) also inherit from parent's commitments, or only from Memory/Knowledge?
   - *Deferred to design phase*
2. **Cross-silo trusted delegation**: Should an agent in silo X ever be allowed to inherit from silo Y in the same org?
   - *Recommendation: No; one silo per org in MVP.*
3. **Agent lifecycle webhook**: Should deletion of a parent trigger cleanup of orphaned children?
   - *Deferred; soft delete is sufficient for v1.5.*
4. **Lineage depth per scenario**: Is 3 hops optimal, or should it be configurable by agent type?
   - *Recommendation: Fixed at 3 for v1.5; revisit post-launch.*
