## Phase 1: Retention Policy

### Problem

Multiple node types grow unboundedly:
- Auto-reflection MetaObservation nodes (cursor grows per supersession/revision)
- Pattern nodes (never pruned)
- Session chains in Intelligence layer (compaction truncates but doesn't delete)
- Low-heat Memory nodes that never promote or decay

The EAG spec promises "decay unless promoted" but no retention enforcement exists.

### Design Questions

1. **What triggers retention?** Options:
   - Time-based (nodes older than X days)
   - Count-based (keep last N per type)
   - Heat-based (below threshold for Y days)
   - Hybrid

2. **What happens to retained nodes?** Options:
   - Hard delete (simple, loses audit trail)
   - Archive to cold storage (keeps audit, adds complexity)
   - Summarize-then-delete (preserves meaning, LLM cost)

3. **Per-silo or global policy?**

4. **Which node types have retention?**
   - Memory: yes (decay_class already hints at this)
   - MetaObservation: yes (unbounded cursor)
   - Pattern: unclear (patterns are derived, maybe just prune stale ones)
   - Intelligence chains: unclear (compaction handles, but originals persist)

5. **Where does retention run?** Dagster asset/sensor or standalone cron?

### Edge Cases (from adversarial review)

1. **Retention + summarization race:** A node scheduled for retention while its chain is being summarized.
   - **Fix:** Retention acquires a lock on the chain before deletion. Summarization checks node existence before writing summary edges.

2. **In-flight queries during deletion:** Query reads a node mid-retention sweep.
   - **Fix:** Soft-delete pattern (tombstone flag) with grace period. Queries skip tombstoned nodes. Hard delete after grace period.

3. **Heat calculation depends on pruned nodes:** Heat signals become unreliable.
   - **Fix:** Heat is recalculated on remaining nodes post-retention, not cached from pre-retention state.

## Phase 2: LLM Summarization for Chains

### Problem

Session compaction (T6) currently truncates chains longer than 5 steps. Information is lost.

### Design Questions

1. **When to summarize vs truncate?**
   - Always summarize (expensive)
   - Summarize if chain > N steps
   - Summarize if chain has high heat

2. **What model/cost budget?**
   - Haiku for cheap bulk summarization
   - Sonnet for important chains
   - Heat-based model selection

3. **Summary granularity?**
   - One summary per chain
   - Hierarchical (summarize segments, then summarize summaries)

4. **Where does summary live?**
   - Replace chain nodes with summary node
   - Summary node with SUMMARIZES edges to originals
   - Archive originals, keep summary active

5. **LLM unavailability:** What if model is rate-limited or down during summarization?

### Edge Cases (from adversarial review)

1. **Summary nodes subject to retention:** Creates recursive pruning.
   - **Fix:** Summary nodes are marked `retention_exempt: true`. They only get pruned when their entire silo is deleted.

2. **Hierarchical summarization scope creep:** Could become a compression subsystem.
   - **Fix:** v1.2 ships single-level summarization only. Hierarchical is v1.3+ if needed.

3. **LLM fallback:** Rate limits or downtime.
   - **Fix:** Fallback to truncation with `summarization_failed: true` flag. Dagster sensor retries failed nodes on next run.

## Phase 3: Auto-Reflection Triggers

### Problem

Auto-reflection currently only fires on supersession/revision. The EAG spec lists additional triggers:
- Confidence shift (delta > threshold)
- Contradiction detected
- High-uncertainty query results

### Design Questions

1. **Confidence shift:** What threshold? Per-node or aggregate?

2. **Contradiction:** How detected? LLM or structural (CONTRADICTS edge)?

3. **High-uncertainty query:** What counts as high-uncertainty? Low result count? Low confidence scores?

4. **Retention for reflections:** Ties back to Phase 1.

### Edge Cases (from adversarial review)

1. **Contradiction detection assumes CONTRADICTS edges exist:** But these aren't always created.
   - **Fix:** Structural-first (check for CONTRADICTS edges), LLM-fallback for unlinked nodes with semantic similarity above threshold.

2. **High-uncertainty query scope creep:** Could become query-time confidence scoring.
   - **Fix:** v1.2 uses simple heuristic: result count < 3 OR average confidence < 0.5. No fancy scoring.

## Phase 4: Config Tuning

### Problem

Four thresholds are hardcoded:
- Belief density threshold (N=3)
- Revision cosine threshold (15%)
- Pattern minimum frequency
- Compaction step threshold (5)

### Design

Surface via:
1. Environment variables (global defaults)
2. Per-silo metadata (override per tenant)
3. MCP admin tool or REST endpoint to update

**Dependency:** Per-silo config mechanism depends on Phase 1's per-silo vs global decision. If Phase 1 chooses per-silo, reuse that pattern here.

## Success Criteria

- [ ] Retention policy prevents unbounded growth for all node types
- [ ] Long chains summarized (preserves semantic meaning, accepts detail loss)
- [ ] Auto-reflection fires on all spec'd triggers
- [ ] All 4 thresholds configurable per-silo
- [ ] Protocol boundary test passes with 0 violations

## Testing Strategy

- Retention: seed a silo with 10k nodes across types, run retention, verify counts bounded
- Summarization: create 10-step chain, compact, verify summary captures key steps
- Auto-reflection: mock confidence shift / contradiction / low-result query, verify MetaObservation created
- Config: set per-silo override, verify behavior changes

## Out of Scope

- Meta-memory hierarchy (v1.3)
- Cross-silo metacognition (v1.3)
- Negative causation PREVENTS edge (v1.3)
- Pattern detection beyond temporal_correlation (v1.3)
